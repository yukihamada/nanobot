## 議題1: Elio (ElioChat)

### CEO 所見
Elioは我々のポートフォリオの中で最もユニークなポジションにいる。「完全オフラインAI」は2026年の市場でほぼ唯一無二だ。GAFAMのAIはすべてクラウド依存。Elioだけがデバイス上で完結する。これは技術的な差別化ではなく、思想的な差別化。プライバシーが社会問題化している今、このポジションは正しい。

昨日（2/23）の成果は地味だが堅実。3バグ修正、150テスト全パス、v1.2.38 build49。App Store再審査に向けた準備が整った。焦ってリリースするより、品質を上げてから出すほうが長期的に正しい。

**判断**: 全力投資を継続。App Store審査通過を最優先。

### CTO 所見
技術的に見て、Elioの成熟度は高い。

**現在の技術スタック**:
- Swift/SwiftUI、iOS 17+/macOS 14+
- 30+のオンデバイスLLMモデル（Qwen3-0.6B, Nemotron日本語版等）
- P2P推論（MultipeerConnectivity / iPhone-Mac間）
- MCP 13サーバー連携
- KokoroTTS / ReazonSpeech（iOS限定）

**昨日修正した3バグ**:
1. テストエラー: Xcode Cloudでのビルド安定化
2. QRコードにElio IDが欠損していた問題
3. addFriend APIのエンドポイント不足

いずれもP2P機能の根幹に関わる修正であり、これが直ったことでフレンド追加フローが初めて完全に動作する。

**技術課題**:
- Mac Catalystのonnxruntime制限（macOSではTTS/STT不可、スタブ対応中）
- LLM推論のメモリ管理（大型モデル使用時のOOM対策）
- App Store審査でのバイナリサイズ（LLMモデル同梱）

**推奨**: onnxruntimeのmacOS対応は後回し。まずiOSのApp Store審査通過に集中すべき。

### CMO 所見
市場ポジションは極めて明確。問題はそれをどう伝えるか。

**競合との差別化マトリクス**:
| | ChatGPT | Gemini | Copilot | **Elio** |
|---|---|---|---|---|
| オフライン | x | x | x | **完全対応** |
| P2P推論 | x | x | x | **対応** |
| デジタル金庫 | x | x | x | **対応** |
| 防災マップ | x | x | x | **対応** |
| 価格 | 有料 | 有料 | 有料 | **完全無料** |

この表を見れば一目瞭然。問題は「誰に」「どう」伝えるか。

**ターゲットユーザー候補**:
1. **プライバシー重視層**: 医療従事者、弁護士、企業の機密情報取扱者
2. **オフライン必要層**: 山岳登山者、海外出張者、災害対策意識の高い層
3. **技術好奇心層**: 自分のデバイスでAIを動かしたい開発者・ギーク

**課題**: App Store審査が通らないとマーケティングは始められない。審査通過後、ProductHunt・Hacker News・日本のテック系メディアへのプレスリリースを準備すべき。

### CPO 所見
プロダクトとしてのElioは、機能は十分すぎるほどある。問題は「最初の5分間の体験」。

**オンボーディングフロー改善提案**:
1. 起動 → モデル自動ダウンロード（バックグラウンド）→ 即座にチャット開始可能に
2. 「機内モードでも動くAI」のデモを初回起動時に実施
3. P2Pフレンド追加のチュートリアル

**v1.3ロードマップ提案**:
- Elio IDの簡素化（QRなしでもID直接入力で追加可能に）
- チャット履歴のiCloud同期（オプション、ユーザー選択制）
- ウィジェット対応（ホーム画面からワンタップでAI質問）

## 議題2: BANTO

### CEO 所見
率直に言う。BANTOは1ヶ月以上停止している。最終コミットは1月20日。バックエンドAPIは完全停止、CI/CDは全敗、ユーザーは0人。

3つの選択肢がある:
1. **復活**: エンジニアリソースを投入して立て直す
2. **凍結**: コードを保存し、市場環境が変わるまで待つ
3. **ピボット**: BANTOのコードベースを別プロダクトに転用する

建設業向け請求書AIは市場としては存在するが、我々のリソースは有限だ。Elioが今まさに審査直前という状況で、BANTOに人を割く余裕はない。

**判断**: 凍結。ただし、コードベース（50+テーブル、35 Edge Functions、133テスト）は資産として保存する。

### CTO 所見
BANTOの技術的状態を正直に報告する。

**アーキテクチャ**:
- React 18 + TypeScript + Vite（フロントエンド）
- Supabase（PostgreSQL + Auth + Storage + 35 Edge Functions）
- 50+テーブル、64マイグレーションファイル
- 133テスト（Vitest + Playwright）

**深刻な問題**:
- Supabase Edge Functionsの複数が応答しない（タイムアウト or 500エラー）
- RLSポリシーの複雑化（テーブル間の依存関係が循環）
- CI/CDパイプラインが全テスト失敗状態
- Node.js依存関係の脆弱性警告多数

**復旧に必要な工数**: 最低2-3週間のフルタイム作業。Edge Functions再デプロイ、RLSポリシー再設計、CI修復、依存関係更新。

**技術的推奨**: 凍結に同意。もし将来復活させるなら、Supabaseの設計をゼロから見直したほうが速い。50テーブルは過剰設計。20テーブル以下に統合すべき。

### CMO 所見
建設業向けSaaSは日本市場で需要はある。しかし:

1. **競合が多い**: MFクラウド請求書、freee、Misoca等の大手が既に市場を押さえている
2. **音声入力の差別化だけでは弱い**: 「声だけで請求書完成」は面白いが、ユーザーがそれを求めているかは検証できていない
3. **ユーザー0人**: プロダクトマーケットフィットの検証すらできていない状態

**推奨**: 凍結。マーケティングリソースはElioに集中すべき。

### CPO 所見
BANTOの最大の問題は「作りすぎたこと」。

50テーブル、35 Edge Functions、133テストという規模は、ユーザー0人のプロダクトとしては明らかに過剰。MVPの段階で検証すべきだった。

**教訓**:
- ユーザーインタビューなしにフル機能を作ってはいけない
- 最初の10人のユーザーを獲得する前に、テーブルは10以下に抑えるべき
- 「技術的に素晴らしい」と「ユーザーに必要」は別

**推奨**: 凍結。BANTOの教訓をElioの開発に活かす。特に「小さく出して、検証して、育てる」サイクルを徹底すること。

# 完全オフラインAIと放置されたSaaS ── ソロ開発者の率直な経営判断

2026年2月24日。月曜日の朝、自分自身と経営会議を開いた。

議題は2つのプロダクト。片方は昨日までバグを潰していた。もう片方は1ヶ月以上コードに触れていない。この2つの運命を決める会議だ。

## 動いているほう: Elio

Elio（ElioChat）は、完全オフラインで動くAIチャットアプリだ。iOS/macOS対応、30以上のLLMモデルをデバイス上で実行でき、インターネット接続は一切不要。P2Pでデバイス同士を直接つなぎ、友人のMacの計算力を借りて推論することもできる。

昨日も3つのバグを修正した。テストのビルドエラー、QRコードのID欠損、フレンド追加APIの不足。地味だが、P2P機能の根幹に関わる修正だった。150テスト全パス、v1.2.38 build49。App Store審査に向けた準備が一歩進んだ。

Elioの差別化は明快だ。ChatGPTもGeminiもCopilotも、クラウドがなければ何もできない。Elioだけが機内モードで動く。飛行機の中で、山の上で、災害時に通信が途絶えた状況で。これは技術的な差別化というより、思想の差別化だ。「あなたのデータは、あなたのデバイスから一歩も出ない」という約束。

## 止まっているほう: BANTO

BANTOは建設業向けの請求書AI。「声だけで請求書完成」がコンセプトだった。

技術的にはそれなりに充実していた。50以上のデータベーステーブル、35のサーバーレス関数、133のテスト。React + TypeScript + Supabaseの堅実なスタック。

問題は、バックエンドが完全に止まっていることだ。Edge Functionsがタイムアウトし、CI/CDは全テスト失敗。最終コミットは1月20日。そしてユーザーは0人。

1ヶ月前に手を離したとき、「来週直そう」と思っていた。来週は来なかった。Elioのバグ修正、App Store審査対応、新機能開発。やることは常にあった。BANTOは静かに息を止めた。

## なぜ片方は動き続け、片方は止まったのか

答えはシンプルだ。**情熱の差**。

Elioには「完全オフラインAI」という、自分が本気で信じているビジョンがある。クラウドに依存しないAIの世界。プライバシーが当然のように守られる未来。このビジョンがあるから、地味なバグ修正も苦にならない。150個のテストを全部通すために夜中までコードを書ける。

BANTOには「建設業の請求書を楽にする」という合理的なビジョンがあった。市場は存在する。需要もある。しかし、自分自身がその問題を切実に感じていなかった。建設業の請求書で困った経験がないのだ。合理的に正しいプロダクトと、魂が入ったプロダクトの違い。

## 過剰設計の代償

BANTOの最大の失敗は「作りすぎたこと」だ。

ユーザー0人のプロダクトに50テーブル。35のEdge Functions。133のテスト。これは品質の証ではなく、過剰設計の証拠だ。最初の10人のユーザーを見つける前に、完璧なアーキテクチャを作ろうとした。誰も使わないテーブルの整合性を、誰も読まないテストで検証していた。

対照的に、Elioは「まず動くものを作り、使いながら直す」サイクルを回している。昨日も実際にP2Pでフレンドを追加しようとして、バグを見つけて、その日のうちに直した。ユーザー体験から逆算した開発と、テーブル設計から始めた開発の差だ。

## 凍結という判断

今日、BANTOを凍結した。

殺すのではない。コードベースは保存する。50テーブルの設計、133のテスト、それらは将来の何かに使えるかもしれない資産だ。しかし今、限られたリソースをBANTOに割く理由はない。

代わりに、すべてのエネルギーをElioに注ぐ。App Store審査を通す。最初のユーザーに届ける。オフラインAIという異端のビジョンを、一人でも多くの人に体験してもらう。

## ソロ開発者へ

もし同じ状況にいるなら、一つだけ伝えたいことがある。

**止める勇気を持ってほしい。**

動かないプロダクトを「いつか直す」と抱え続けることは、動いているプロダクトのリソースを奪っている。凍結は失敗ではない。集中のための戦略的判断だ。

ElioのApp Store審査がどうなるかは分からない。しかし少なくとも、今日この瞬間、全力を注ぐプロダクトは一つに絞れた。それだけで、今日の経営会議には意味があった。
